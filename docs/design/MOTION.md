# MOTION.md

## Motion Philosophy
Motion should improve clarity, feedback, and continuity.
It should not exist for spectacle.

## Timing
- Fast for controls and direct manipulation
- Slightly softer for surface transitions
- Avoid long delays in utility flows

## Easing
Choose easing that feels smooth and intentional.
Avoid theatrical or bouncy motion unless the product context strongly supports it.

## Enter and Exit
Entry and exit transitions should support orientation and hierarchy.
They should feel clean, not dramatic.

## Hover, Press, and Focus Feedback
Interaction feedback should be noticeable but restrained.

## Loading Motion
Loading motion should reassure, not distract.
If a static or simpler loading state communicates better, prefer that.

## Page and Section Transition
Transitions should help maintain context between related surfaces.
Avoid complex movement that makes the interface feel slower.

## Reduced Motion Rules
Support reduced motion preferences.
When motion is reduced, preserve clarity through opacity, state change, and layout stability.

## Motion Don'ts
- Do not animate every state change.
- Do not add motion that delays task completion.
- Do not use animation to compensate for weak hierarchy.
