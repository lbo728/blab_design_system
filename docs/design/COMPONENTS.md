# COMPONENTS.md

## Component Philosophy
Components should feel consistent, reusable, and intentionally composed.
BLab components should solve common UI needs without encouraging visual improvisation.

## Shared Component Rules
- Reuse token hierarchy consistently.
- Preserve predictable spacing, radius, and state behavior.
- Favor calm emphasis over exaggerated styling.
- Keep interactive affordances obvious.
- Design disabled, focus, error, and loading states intentionally.

## Buttons
Buttons should express hierarchy clearly.
Only the most important action should carry the strongest emphasis.

## Cards
Cards should group content with structure and calm depth.
Prefer surface and tone differences before heavy borders.

## Text Fields
Inputs should prioritize readability, focus clarity, validation visibility, and comfortable spacing.

## Chips and Tags
These should support filtering, categorization, and compact metadata without becoming visually noisy.

## Modals and Sheets
Overlays should feel deliberate and not interruptive beyond necessity.
They should preserve hierarchy, focus, and readability.

## Navigation Elements
Navigation should communicate structure simply.
Do not overload navigation with competing visual emphasis.

## Empty State Blocks
Empty states should combine clarity, light encouragement, and helpful next actions.

## Feedback Components
Toasts, banners, helper text, and alerts should prioritize message clarity over decoration.
