# PATTERNS.md

## Pattern Philosophy
Patterns define how multiple components work together.
BLab patterns should reduce decision fatigue and increase structural consistency across products.

## Forms
Forms should emphasize clarity, progression, validation feedback, and low friction.

## Lists
Lists should be easy to scan, sort, and act upon.
Hierarchy should come from spacing, grouping, and typography before heavy chrome.

## Detail Views
Detail surfaces should balance overview and depth without overwhelming users.

## Empty States
Empty states should explain the situation, reduce uncertainty, and suggest a meaningful next step.

## Loading States
Loading patterns should communicate progress with calm and readable feedback.
Avoid excessive animation.

## Error States
Errors should be specific, actionable, and visually distinct without becoming alarming by default.

## Confirmation Flows
Use confirmation only when the action has meaningful consequence.
Do not add confirmation friction to low-risk actions.

## Search and Filter Patterns
Search and filtering should feel efficient, composable, and easy to reset.

## Dashboard Patterns
Dashboards should prioritize glanceability, grouping, and decision support.

## Onboarding Patterns
Onboarding should reduce ambiguity and build confidence quickly.
